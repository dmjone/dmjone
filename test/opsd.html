Section 1

For the Venture Capitalist: What this is & why now

The system: a SaaS authenticity layer for business‑critical PDFs. Every document is signed server‑side with a standards‑compliant PAdES signature (CMS/CAdES) and optionally timestamped via RFC 3161. Long‑Term Validation (LTV) data are embedded (DSS/VRI), so a recipient can validate authenticity offline for years.

Why people need it: PDFs are the de facto carrier of invoices, certificates, statements, results, and policies. Without cryptographic binding, they’re easy to copy or alter. Our approach makes tampering detectable in standard readers and enables immediate revocation if a mistake or compromise is found.

How it helps organizations: zero friction for receivers (works in Adobe/OS viewers), provable integrity for auditors, and clean server‑side controls (key custody, issuance, revocation, registry, and audit). Timestamping anchors documents to trusted time and LTV preserves verifiability beyond certificate lifetimes.

Go‑to‑market: compliance (policies, pay‑slips), education (transcripts), healthcare (lab results), fintech (statements), logistics (e‑CoC), and public sector certificates. The win is standards interoperability—no proprietary viewer.
What this does for a VC
  • Product moat from standards + PKI rather than lock‑in.
  • Horizontal need across verticals (high reuse, low churn).
  • Operational leverage: Cloudflare Workers + D1 keep infra lean.
  • Clear unit economics—per‑document issuance & verification at web scale.

Technologies, Standards & Jargon (accurate to code)

Core runtimes & services
  • Cloudflare Workers (edge execution) + D1 (SQLite‑compatible DB) for the admin portal, registry, and audit.
  • NGINX for static PKI publishing (AIA/CDP) and reverse‑proxy to signer & OCSP.
  • OpenSSL CA/OCSP/TSA components (internal PKI, CRLs, OCSP responses, RFC 3161 server).
Java Signer microservice
  • Language: Java 21 (OpenJDK) + Maven.
  • Web framework: Javalin 6.x (HTTP API).
  • PDF engine: Apache PDFBox 3.0.6 for signing, DSS editing, and verification.
  • Crypto: Bouncy Castle (bcprov/bcpkix 1.82) for CMS/CAdES, TSP, OCSP helpers.
  • JSON & logging: Jackson Databind 2.18.x, SLF4J 2.0.x.
Open standards (what the code implements)
  • PAdES (ETSI EN 319 142) — PDF Advanced Electronic Signatures, profiles B‑T/B‑LT/B‑LTA.
  • CMS/CAdES containers, with SigningCertificateV2 / ESSCertIDv2 (RFC 5035).
  • RFC 3161 timestamping for signature‑time‑stamp and document time‑stamp (“DocTimeStamp”).
  • OCSP (RFC 6960) and CRL (RFC 5280) for status and LTV materials.
  • PDF ByteRange covers whole document except /Contents for PAdES & DocTimeStamp.
Security & transport
  • HMAC‑SHA‑256 headers (x-worker-hmac, x-worker-ts, x-worker-nonce) gate signer APIs.
  • Ephemeral per‑document certificate issuance (leaf RSA‑3072, chain to issuing CA) + CRL refresh.
  • Admin session cookies are HMAC‑sealed (edge‑verified); CSRF: SameSite/Strict + double‑submit token.

Section 3

How it works: Signing • Revocation • Verification

A) Signing (server‑side, deterministic)
  1. Admin uploads a PDF through the Cloudflare Worker admin portal (edge).
  2. Worker computes HMAC over method+path+timestamp+nonce+body and calls the signer’s /bundle.
  3. Signer generates a one‑off document certificate (RSA‑3072) from the issuing CA and signs the PDF with subFilter = ETSI.CAdES.detached (PAdES). (PDFBox + Bouncy Castle)
  4. Signer optionally adds RFC 3161 signature‑time‑stamp and a full‑document DocTimeStamp, then embeds DSS/VRI (chain + CRL [+ OCSP if reachable]) for LTV.
  5. Signer self‑verifies the result (ByteRange covers document; CMS validates; issuer chain matches) and publishes a ZIP containing signed.pdf and the Trust Kit at pki.*.

Why this matters: recipients can verify in Acrobat/Reader offline for years (LTV).

B) Revocation (instant kill‑switch)
  1. Admin triggers revoke; Worker HMAC‑calls signer /revoke with the certificate serial.
  2. Signer marks the cert revoked in the issuing CA database and regenerates the CRL; OCSP reflects status within minutes.
  3. Any subsequent validation (in Acrobat or our verifier) shows the certificate as revoked.
C) Verification (user or machine)
  • Drag‑and‑drop PDF into our public verifier (Worker → Signer /verify), or open in Acrobat.
  • Checks performed: signature object exists; CMS cryptography valid; ByteRange covers the document; signer chain anchors to our CA; CRL/OCSP not revoked; issuer matches our key (SPKI).
  • Outcome: VALID (genuine and current) or TAMPERED/REVOKED with clear reasoning.
Section 4

System Data Flow (end‑to‑end)

A) Admin signs a document
Admin (browser)
   │ 1. Upload PDF + meta
   ▼
Cloudflare Worker (edge)
   │ 2. HMAC(ts, nonce, method, path, body)
   │ 3. Store audit → D1
   ▼
Java Signer (Javalin)
   │ 4. OpenSSL CA issues per‑doc cert (RSA‑3072)
   │ 5. PDFBox + BC → PAdES (CAdES.detached), ESSCertIDv2, RFC3161 stamp
   │ 6. Embed DSS/VRI (chain + CRL [+ OCSP]) for LTV
   │ 7. Self‑verify (ByteRange, CMS, issuer)
   │ 8. Publish ZIP { signed.pdf + Trust Kit } → pki.*
   ▼
Worker returns direct PKI URL to Admin + registry row in D1
B) Revocation & status
Admin → Worker → Signer /revoke(serial)
Signer:
  - Mark revoked in CA index
  - Regenerate CRL (publish at AIA/CDP)
  - OCSP responder answers "revoked"
Verifiers (Acrobat / our verifier) pick up CRL/OCSP freshness
C) Public verification
User → Verifier:
  - Compute SHA‑256; call signer /verify
  - PDFBox parses PDSignature(s); Bouncy Castle validates CMS
  - Check ByteRange coverage, signer chain, revocation (CRL/OCSP)
Result → VALID or TAMPERED/REVOKED (+ issuer DN, subFilter, facts)
