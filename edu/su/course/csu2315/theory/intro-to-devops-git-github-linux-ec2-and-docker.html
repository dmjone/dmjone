1. Introduction to DevOps

1.1 Definition

DevOps is a set of cultural practices, processes, and tools that bring Development and Operations together to deliver software quickly, safely, and repeatedly. It emphasizes automation, CI/CD, infrastructure as code, and continuous feedback.

1.2 Key ideas and vocabulary

  • Goals: short lead time, high deployment frequency, low change-failure rate, fast recovery.
  • CALMS: Culture, Automation, Lean, Measurement, Sharing.
  • Software value stream / pipeline: Plan → Code → Build → Test → Package → Release → Deploy → Operate → Monitor → Feedback.
  • CI (Continuous Integration): every code change triggers automatic build and tests.
  • CD (Continuous Delivery/Deployment):
    • Delivery: every change is deployable after CI.
    • Deployment: changes auto-deploy to production after checks.
  • IaC (Infrastructure as Code): infrastructure declared in files and versioned (e.g., Terraform/CloudFormation).
  • Release strategies: Blue/Green (two environments, switch traffic), Canary (small percentage first), Rolling (gradual replacement).
  • Observability: logs, metrics, traces; SLA/SLO/SLI relations.

1.3 One-minute practical narrative

A small web app is pushed to GitHub; a CI workflow builds and tests on each push; a Docker image is built and stored in a registry; CD deploys the container to an EC2 instance; monitoring collects logs/metrics; feedback informs the next change.

2. Git and GitHub

2.1 Definitions

  • Git: a distributed version control system that saves code as commits (snapshots), supports branches, merges, and history.
  • GitHub: a cloud platform that hosts Git repositories and adds collaboration (pull requests, issues, Actions for CI).
  • Commit: a snapshot with a message and metadata.
  • Branch: an independent line of commits; HEAD points to the current commit.
  • Remote: a repository hosted elsewhere (e.g., origin on GitHub).

2.2 Mental model

Workflow moves through three areas: Working directory → Staging area (index) → Repository.

2.3 Initial configuration (git config)

git config sets and reads configuration values at three scopes: local (current repo), global (current user), system (machine). Local overrides global; global overrides system.

# Identity (once per machine)
git config --global user.name "Your Name"
git config --global user.email "you@example.com"

# View and edit configuration

git config --list --show-origin
git config --get user.email
git config --unset --global user.email
git config -e --global                     # edit global config in editor

# Useful options

git config --global init.defaultBranch main
git config --global core.editor "code --wait"
git config --local core.ignorecase false

2.4 Create a repository and first push to GitHub

mkdir demo && cd demo
git init
echo "# Demo" > README.md
git add .
git commit -m "first commit"
git branch -M main

# After creating an empty repo on GitHub named demo

git remote add origin [https://github.com/<user>/demo.git](https://github.com/<user>/demo.git)
git push -u origin main

2.5 Everyday commands

git status
git add file.txt           # or: git add .
git commit -m "meaningful message"
git log --oneline --graph --decorate
git show <commit-id>
git diff                   # unstaged vs working tree
git diff --staged          # staged vs last commit

2.6 Branching, merging, and collaboration

git switch -c featureX     # create and move to feature branch
# ...edit files...
git add . && git commit -m "feat: add X"
git switch main
git merge featureX         # fast-forward or merge commit

git remote -v
git fetch                  # download new objects/refs
git pull                   # fetch + merge current branch
git push                   # upload to remote tracking branch

2.7 Undo and investigation

git restore file.txt                 # discard unstaged changes
git restore --staged file.txt        # unstage
git revert <commit>                  # safe undo (creates new commit)
git reset --hard <commit>            # move HEAD & overwrite files (use cautiously)
git reflog                           # locate lost commits/HEAD movements
git blame filename.ext               # per-line history

  • Permissions: three roles—user, group, others—and three bits—read(4), write(2), execute(1).

4. AWS EC2

4.1 Definitions

  • Amazon EC2: virtual machines in AWS with configurable CPU, memory, storage, and networking.
  • AMI: an OS image used to launch instances.
  • Security Group: a stateful virtual firewall controlling inbound/outbound ports.
  • EBS volume: network-attached persistent storage for EC2 instances.
  • Elastic IP: a static public IPv4 address that can be attached to an instance.

4.2 Launch checklist (console)

  1. Choose AMI (e.g., Ubuntu 22.04 or Amazon Linux 2).
  2. Select instance type (e.g., t2.micro/t3.micro).
  3. Create or pick Key Pair (.pem).
  4. Configure Security Group (allow SSH/22 from your IP; add HTTP/80 or HTTPS/443 for a website).
  5. Accept storage defaults (EBS) and launch.

4.3 Connect via SSH

chmod 400 mykey.pem
# Ubuntu:
ssh -i mykey.pem ubuntu@<PUBLIC_IP>
# Amazon Linux:
ssh -i mykey.pem ec2-user@<PUBLIC_IP>

4.4 Simple user-data script (runs once at first boot)

#!/bin/bash
apt update -y
apt install -y nginx
systemctl enable --now nginx
echo "Hello from EC2" > /var/www/html/index.html

4.5 Operational notes

  • Stop: compute is halted; EBS volumes persist.
  • Terminate: instance is deleted; root EBS often deleted (depending on setting).
  • Prefer IAM roles over static keys for applications running on EC2.

4.6 Troubleshooting checklist

  • Key file has permission 400.
  • Correct username (ubuntu or ec2-user).
  • Security group allows port 22 (and 80/443 for web).
  • Instance has a public IP or an attached Elastic IP and resides in a public subnet.

4.7 Practical example (serve a page)

# SSH in, then:
sudo apt update -y
sudo apt install -y nginx
echo "EC2 up" | sudo tee /var/www/html/index.html

5. Docker Basics

5.1 Definitions

  • Docker: a platform to build, ship, and run applications using containers.
  • Image: a read-only template consisting of layers.
  • Container: a running instance of an image, isolated from the host.
  • Dockerfile: a text file describing how to build an image.
  • Registry: a service that stores and distributes images (Docker Hub/ECR).

5.2 Core commands

docker pull ubuntu
docker images
docker run -it --name myub ubuntu bash      # create + start + attach
exit
docker ps                                    # running
docker ps -a                                 # all
docker start -ai myub                        # start and attach
docker exec -it myub bash                    # run command inside
docker stop myub
docker rm myub
docker rmi <image-id-or-name>
docker logs <container-name>

5.3 Build and run your own image (Node example)

hello.js

console.log("Hello from Docker!");

Dockerfile

FROM node:18
WORKDIR /app
COPY . /app
CMD ["node", "hello.js"]

Build and run

docker build -t hellojs:1.0 .
docker run --rm hellojs:1.0

5.4 Publish to Docker Hub

docker login
docker tag hellojs:1.0 <hub-user>/hellojs:1.0
docker push <hub-user>/hellojs:1.0

5.5 Ports, volumes, and common flags

docker run -d --name web -p 8080:80 nginx      # host:container
docker run -it -v $(pwd)/data:/data ubuntu     # bind mount
# ENTRYPOINT vs CMD:
# ENTRYPOINT defines the executable; CMD sets default args (or command if no ENTRYPOINT).

5.6 Practical example (quick web server)

docker run -d --name web -p 8080:80 nginx
# visit http://localhost:8080
docker logs web | tail
docker stop web && docker rm web

5.7 Notes on correctness

  • Use a .dockerignore (e.g., node_modules/, *.log, .git) to reduce build context.
  • -it is for interactive terminals.
  • docker run creates a new container each time unless you reuse --name.

6. End-to-End Practical: Code → Image → Registry → EC2

  1. Create app + Dockerfile (as in §5.3).
  2. Build image: docker build -t hellojs:1.0 .
  3. Push to Docker Hub: docker login, docker tag, docker push.
  4. Launch EC2 (Ubuntu), open ports 22 and 80.
  5. SSH to EC2 and install Docker: 
    sudo apt update -y
sudo apt install -y docker.io
sudo systemctl enable --now docker
sudo usermod -aG docker ubuntu   # log out/in to apply
  6. Run container on EC2: 
    docker pull <hub-user>/hellojs:1.0
docker run --rm <hub-user>/hellojs:1.0
# if it were a web app on port 3000:
# docker run -d -p 80:3000 <hub-user>/webapp:1.0

This exercise connects all five topics and provides a clear demonstration of DevOps flow.

8. Exam-Style Very Short Question Bank with Model Content

8.1 Two-mark questions (short, direct)

  1. Define DevOps.

    DevOps unifies development and operations through automation, CI/CD, and shared ownership to deliver software quickly and reliably.

  2. What is Continuous Integration?

    Automatic build and tests triggered by each change to keep the main branch always healthy.

  3. Differentiate image and container.

    Image is a read-only template; container is a running instance of that template.

  4. Purpose of a Security Group.

    A stateful virtual firewall controlling inbound and outbound traffic to an EC2 instance.

  5. Meaning of chmod 755.

    Permissions become rwx r-x r-x for user, group, others.

  6. git pull vs git fetch.

    pull = fetch + merge into current branch; fetch only downloads.

  7. Define Continuous Delivery.

    Every change is automatically built and tested and kept in a deployable state, awaiting an approval or trigger to release.

  8. Define Continuous Deployment.

    Every change that passes automated tests is automatically deployed to production without a manual gate.

  9. What is Infrastructure as Code (IaC)?

    Infrastructure is described in files and versioned like code, giving repeatability and fewer manual errors.

  10. What is a Git commit?

    A saved snapshot of the repository with metadata and a message describing the change.

  11. What is a Git branch?

    An independent line of development that lets you work without affecting the main branch.

  12. What does git status show?

    Current branch, staged/unstaged changes, and untracked files relative to the last commit.

  13. Purpose of .gitignore.

    Lists patterns of files and folders Git should not track (e.g., node_modules, logs, secrets).

  14. What is HEAD in Git?

    A pointer to the current commit; usually the tip of the currently checked-out branch.

  15. What is a merge conflict?

    A situation where Git cannot auto-merge changes to the same lines and needs manual resolution.

  16. Use of git revert.

    Creates a new commit that undoes a previous commit; safe on shared branches.

  17. What is an AMI?

    An Amazon Machine Image that contains the OS and software used to launch EC2 instances.

  18. What is an EC2 key pair used for?

    Public/private keys for SSH authentication to log in securely to an instance.

  19. What is EC2 user data?

    A boot-time script that runs once at first launch to install or configure software automatically.

  20. Define Elastic IP.

    A static public IPv4 address that you can attach to and move between EC2 instances.

  21. What is an EBS volume?

    Persistent block storage attached to EC2 instances; survives stop/start of the instance.

  22. Define Dockerfile.

    A text recipe describing how to build a Docker image (base image, files, commands).

  23. Meaning of -p host:container in Docker.

    Publishes a container port to a host port (e.g., -p 8080:80 exposes container 80 on host 8080).

  24. What is a Docker volume?

    Persisted storage or bind-mounted host path used to keep or share data across container runs.

  25. What does docker ps -a display?

    All containers on the host, including running and stopped ones.

  26. Difference between docker run and docker exec.

    docker run creates and starts a new container; docker exec runs a command inside an existing container.

  27. Define Continuous Testing.

    Automated tests run throughout the pipeline to give fast feedback on quality and risk for every change.

  28. Define Pull Request (PR).

    A request to merge changes from one branch into another, enabling code review and automated checks.

  29. What is a Git remote?

    A reference to a repository hosted elsewhere (e.g., GitHub) used for fetching and pushing.

  30. What is a fast-forward merge?

    A merge where the target branch pointer simply moves forward because no divergent commits exist.

  31. What is rebase used for?

    To replay commits onto a new base, creating a linear history.

  32. Purpose of .dockerignore.

    Excludes files from the build context to speed builds and avoid packaging unnecessary content.

  33. What does tail -f do?

    Shows the end of a file and follows new lines as they are appended (live log viewing).

  34. What does ss -tulpen show?

    Lists listening and established sockets with protocols, ports, processes, and PIDs.

  35. EC2: stopped vs terminated.

    Stopped halts compute but keeps EBS storage; terminated deletes the instance (root volume usually removed).

  36. What is an IAM role for EC2?

    A temporary-credentials identity attached to an instance so apps can access AWS APIs without hard-coded keys.

8.3 Five-mark questions (complete steps + brief explanation)

A. Build an image, push to Docker Hub, and run it.

# Dockerfile
FROM node:18
WORKDIR /app
COPY . /app
CMD ["node", "hello.js"]

docker build -t hellojs:1.0 .
docker login
docker tag hellojs:1.0 <hub-user>/hellojs:1.0

docker run --rm <hub-user>/hellojs:1.0

Explanation: Dockerfile defines base, working directory, file copy, and start command; image is built, tagged, pushed, and executed.

B. Launch a web server on EC2 and display a page.

  1. Launch Ubuntu t2.micro; Security Group allows 22 and 80; create a key pair.
  2. Connect: 
    chmod 400 key.pem
ssh -i key.pem ubuntu@<PUBLIC_IP>
  3. Install and start Nginx: 
    sudo apt update -y
sudo apt install -y nginx
echo "Hello EC2" | sudo tee /var/www/html/index.html
  4. Access http://<PUBLIC_IP>. Mention that SG is the firewall and the default user is ubuntu.

C. Containerize a simple Node web server and expose it on host port 8080.

// server.js
const http = require('http');
const port = 3000;
http.createServer((req,res)=>{res.end('Hello Web');}).listen(port); 
# Dockerfile
FROM node:18
WORKDIR /app
COPY . /app
CMD ["node", "server.js"]
docker build -t webjs:1.0 .
docker run -d -p 8080:3000 webjs:1.0

Explanation: the image contains Node and the app; port mapping exposes container 3000 on host 8080.

D. Full Git workflow: clone, branch, change, push, and open a PR.

git clone https://github.com/<user>/repo.git
cd repo
git switch -c feature/update-readme
echo "Study rule: commit often." >> README.md
git add README.md
git commit -m "feat: add study rule"
git push -u origin feature/update-readme

Explanation: push creates the remote branch; open the repository on GitHub and create a Pull Request from the feature branch to main.

E. Find error lines in logs, compress, and transfer to a remote server.

cd /var/log
grep -R "ERROR" . > ~/errors.txt
tar -czf ~/errors.tgz ~/errors.txt
scp -i key.pem ~/errors.tgz ubuntu@<PUBLIC_IP>:/home/ubuntu/

Explanation: searches recursively, archives results, and securely copies the archive to a server.

F. Use EC2 user data to auto-install and start Nginx at launch.

#!/bin/bash
apt update -y
apt install -y nginx
systemctl enable --now nginx
echo "Boot-installed by user data" > /var/www/html/index.html

Explanation: paste this script in the User data field when launching the instance; after boot, visit http://<PUBLIC_IP> to verify.

G. Demonstrate persistent data with a Docker bind mount.

mkdir -p ~/data
docker run -it --name demo -v ~/data:/data ubuntu bash -lc 'echo persisted > /data/file.txt'
docker rm demo
docker run --rm -it -v ~/data:/data ubuntu bash -lc 'cat /data/file.txt'

Explanation: the file survives container deletion because it is stored on the host via the mounted path.

H. Add a minimal GitHub Actions CI to run Node tests on pushes and PRs.

# .github/workflows/ci.yml 
name: CI
on: [push, pull_request]

  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4

          node-version: '18'
      - run: npm ci
      - run: npm test -- --ci

git add .github/workflows/ci.yml
git commit -m "ci: add node tests"
git push

I. Create a multi-stage Dockerfile (build + runtime) and run it.

# Dockerfile

docker run -d --name static -p 8080:80 -v $(pwd)/site:/usr/share/nginx/html:ro nginx

Explanation: the host directory is mounted read-only into Nginx’s web root and served on http://localhost:8080.

git reflog

N. Squash the last 3 commits into one and update the remote.

git rebase -i HEAD~3 # mark the last two commits as 'squash' or 's'

O. Run a container with CPU and memory limits and view its stats.

9. Appendices

9.1 Linux quick list (one-liners)

  • pwd (print path), ls -la (list with details), cd (change dir), mkdir -p (nested), rm -rf (recursive delete), grep -R (recursive search), find (locate files), df -h (disk free), du -sh * (sizes), ss -tulpen (ports), tar -czf/-xzf (archive/compress).

9.2 Git quick list

  • init, status, add, commit, log --oneline --graph, switch -c, merge, remote add, push -u, pull, fetch, revert, reset --hard, restore, reflog, blame.

9.3 Docker quick list

  • pull, images, run -it, ps/ps -a, start -ai, exec -it, stop, rm, rmi, logs, build -t, tag, login, push, run -p, run -v.

9.4 EC2 quick list

  • AMI, instance type, key pair, security group, EBS volume, Elastic IP, SSH users (ubuntu/ec2-user), user-data script, stop vs terminate.
