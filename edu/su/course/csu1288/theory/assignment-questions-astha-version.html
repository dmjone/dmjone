1. Installation and Configuration of Snort on Ubuntu (10 Marks)**

Concept: Snort is an open-source network IDS/IPS that captures and inspects packets in real-time to detect attacks. Steps:

  1. Update system:
sudo apt update && sudo apt upgrade
  1. Install Snort and dependencies:
sudo apt install snort libpcap-dev libpcre3-dev zlib1g-dev

During install, specify the network interface (e.g. eth0) and your HOME_NET (e.g. 192.168.1.0/24). 3. Configure Snort:

  • Edit /etc/snort/snort.conf
var HOME_NET 192.168.1.0/24  
include $RULE_PATH/local.rules
  • Ensure rule paths and variables point to /etc/snort/rules.
  1. Enable and start service:
sudo systemctl enable snort
sudo systemctl start snort
  1. Test configuration:
sudo snort -T -c /etc/snort/snort.conf

Expected: “Configuration validated with 0 errors.” 6. Run live:

sudo snort -A console -q -c /etc/snort/snort.conf -i eth0

Example: After installation, sudo snort -T -c /etc/snort/snort.conf returns no errors, confirming correct setup.

2. Rule Headers (10 Marks) Concept: The rule header defines what traffic to inspect by specifying action, protocol, addresses, ports, and direction. Structure: 

action protocol src_ip src_port direction dst_ip dst_port
  • action: alert, log, pass, drop, reject, activate
  • protocol: tcp, udp, icmp, ip
  • src_ip/src_port: IP (or variable like $EXTERNAL_NET), port number (or name like http, any)
  • direction: -> (unidirectional) or <-> (bidirectional)
  • dst_ip/dst_port: same format as source

Example:

alert tcp $EXTERNAL_NET any -> $HOME_NET 22

This header alerts on any TCP packet from external hosts to port 22 on the home network.

3. Rule Options (10 Marks) Concept: Rule options, enclosed in (...), define how to inspect packets that match the header. They consist of key:value pairs for message, matching patterns, classification, etc. Common Options:

  • msg: description of the alert
  • sid: unique Snort rule ID
  • rev: revision number
  • content: byte or string pattern to match
  • classtype: threat category (e.g. trojan-activity)
  • reference: external links (e.g. url,example.com)
  • threshold: rate-limit alerts

Example:

alert tcp $EXTERNAL_NET any -> $HOME_NET 80 (msg:"WEB-MISC Bad User-Agent"; content:"BadBot"; sid:1000001; rev:1;)

This alerts when an HTTP request to port 80 contains the string “BadBot” in its payload.

4. Snort Modules (10 Marks) Concept: Modules extend Snort’s core detection engine and are configured in snort.conf. They preprocess traffic, apply detection plugins, and output alerts.

  • Preprocessors: Normalize or reassemble traffic before detection.

    • stream5_global: reassembles TCP streams
    • http_inspect: parses HTTP protocol

  • Detection Plugins: Core engine that applies the header and options to packet data.

  • Output Plugins: Format and log alerts.

    • alert_fast: simple alert format
    • unified2: binary format for analysis tools

Example configuration:

preprocessor stream5_global: track_tcp yes, track_udp yes
preprocessor http_inspect: global iis_unicode_map unicode.map 1252
output unified2: filename snort.u2, limit 128

This reassembles TCP sessions, decodes HTTP traffic, and logs alerts in Unified2 format for external analysis tools.

Windows Malware

  • Concept: Windows is the primary target due to its large user base. Malware uses Portable Executable (PE) files (.exe, .dll) and exploits Windows APIs and services. Common infection vectors are phishing emails, malicious websites (drive-by downloads) and USB drives. Elevation often uses UAC-bypass or service-exploitation.

  • Example:

    • WannaCry: Ransomware that spread via the SMBv1 protocol, encrypting files and demanding payment.
    • Emotet: Banking trojan delivered by malicious email attachments; it downloads additional payloads.
    • Conficker: Worm exploiting a Windows Server service vulnerability to form large botnets.

Linux Malware

  • Concept: Linux malware is less widespread but focuses on servers, cloud hosts and IoT devices. It uses ELF binaries and shell scripts. Common entry points are weak SSH credentials, unpatched server software or compromised package repositories. Privilege escalation targets root via kernel or SUID exploits.

  • Example:

    • Mirai: IoT botnet that scans for devices with default SSH/Telnet credentials, then installs an ELF binary to launch DDoS attacks.
    • Linux.Rex: A combined crypto-miner, rootkit and backdoor delivered as an ELF executable on compromised servers.
    • Tsunami: A backdoor Trojan that opens a remote shell by exploiting insecure services.

Android Malware

  • Concept: Android malware targets smartphones and tablets. It is packaged as APKs and exploits the permission-based sandbox (e.g., SMS, contacts, camera). Distribution is via malicious apps in unofficial stores or repackaged popular apps. Root exploits can bypass sandboxing.

  • Example:

    • Anubis: A banking trojan that overlays fake login screens to steal credentials; installed via malicious APKs.
    • Joker: SMS-fraud trojan that auto-subscribes users to premium services without consent.
    • Triada: Sophisticated modular rootkit that injects into system processes to hide additional payloads.

Common Signs of Intrusion in IT Networks

  • Unusual Network Traffic: Sudden spikes in inbound or outbound packets, unexpected protocols or ports in use.
  • Repeated Failed Logins: High volume of authentication failures, indicating brute-force attempts.
  • New or Unexpected Services: Unknown processes listening on ports or scheduled tasks created without authorization.
  • File Integrity Changes: Modified, deleted or added critical system files or configuration files.
  • Privilege Escalation Attempts: Creation of new admin accounts or changes to group memberships.
  • Lateral Movement Indicators: Internal scanning tools running between hosts, unexpected use of remote-desktop or SMB.
  • IDS/IPS Alerts: Signature-based or anomaly alerts for known attack patterns (e.g. port scans, exploit attempts).
  • Unusual CPU/Disk/Memory Usage: Resource exhaustion attacks or hidden malware consuming system resources.

Common Signs of Intrusion in OT Networks

  • Unexpected Control Commands: PLCs or RTUs receiving commands outside normal operational schedules.
  • Sensor/Actuator Anomalies: Sudden jumps in sensor readings or actuators moving without operator input.
  • Protocol Misuse: Use of SCADA/Modbus/DNP3 functions that aren’t part of the process logic.
  • Process Disruptions: Unplanned shutdowns, emergency stops or equipment operating outside set parameters.
  • Time Synchronization Errors: OT devices with drifted clocks can indicate tampering with NTP or internal clocks.
  • Unauthorized Firmware Changes: Firmware on controllers updated without maintenance windows or signatures.

Why OT Intrusion Detection Is More Challenging

  1. Legacy and Proprietary Protocols: Many ICS/SCADA protocols lack encryption or standard definitions, making deep-packet inspection difficult.
  2. Real-Time and Availability Requirements: OT systems cannot tolerate latency; heavy monitoring or active probes risk disrupting critical processes.
  3. Device Resource Constraints: PLCs and RTUs often lack capacity for endpoint agents or advanced security software.
  4. Limited Baseline Data: OT environments vary greatly by process; establishing “normal” behavior requires custom profiling.
  5. Safety-First Mindset: False positives may trigger shutdowns, causing significant physical and economic impact, so thresholds are set high, reducing sensitivity.
  6. Network Segmentation and Air-Gapping: While improving security, segmentation can also limit visibility for centralized monitoring.

1. Static Analysis (10 Marks)

Concept: Examination of a malware sample without executing it. Analysts inspect file structure, code, strings and metadata to understand functionality. Example: Using a disassembler (e.g. IDA Pro) to view assembly code and extract suspicious strings embedded in an EXE. Advantages:

  • Fast initial triage without risk of infection.
  • Can reveal obfuscated code patterns, embedded URLs and configuration data.
  • No need for a live environment or special hardware. Limitations:
  • Cannot observe runtime behavior (e.g. unpacking, memory-only actions).
  • Obfuscation, encryption and packers hinder inspection.
  • May miss network activity or dynamic code. Effective Scenario: When handling untrusted samples safely and extracting IOCs (Indicators of Compromise) like file hashes or hard-coded URLs before deeper testing.

2. Dynamic Analysis (10 Marks) Concept: Running malware in a controlled “sandbox” or virtual machine to observe its real-time behavior, including file, registry and network actions. Example: Executing a suspicious DLL in Cuckoo Sandbox and logging API calls, file writes and outbound connections to identify its payload. Advantages:

  • Reveals true behavior: registry changes, dropped files, spawned processes.
  • Captures network indicators like C2 domains and protocols.
  • Uncovers in-memory unpacking or self-modifying code. Limitations:
  • Risk of sandbox evasion: malware may detect virtual environments and alter behavior.
  • Requires safe, isolated infrastructure and effort to revert snapshots.
  • Time-consuming for large malware families or slow-starting samples. Effective Scenario: When you need to observe actual attack vectors and runtime indicators, especially for sophisticated packers or polymorphic malware.

3. Hybrid Analysis (10 Marks) Concept: Combines static and dynamic methods to leverage strengths of both: initial code inspection guides targeted execution monitoring, and runtime logs feed back into deeper static review. Example: First unzip a packed sample statically to extract a second-stage ELF, then execute it in a Linux sandbox to log system calls, and finally re-disassemble based on observed behavior. Advantages:

  • Comprehensive coverage: catches obfuscated routines and runtime-only code paths.
  • Correlates static IOCs with dynamic events for higher confidence.
  • Reduces manual effort by focusing on hotspots identified dynamically. Limitations:
  • More resource-intensive: requires both analysis environments and skilled analysts.
  • Complexity in correlating static and dynamic data can slow turnaround.
  • May still miss evasion if both environments are detectable. Effective Scenario: When analyzing high-risk or advanced persistent threat (APT) samples where precision is critical and both code structure and live behavior must be understood.

Machine Learning in Intrusion Detection for IT and OT Networks (10 Marks)

Concept: Machine learning (ML) builds models that learn “normal” vs. “malicious” patterns from data. In intrusion detection, ML can classify traffic or sensor readings to spot attacks without manually writing rules.

1. Application in IT Networks

  • Data Sources: Network‐flow records (packet counts, durations, bytes), host logs (login attempts, file changes).
  • Process:
  1. Feature Extraction: Compute features per flow or per host event (e.g. average packet size).
  2. Model Training: Use labelled data (benign vs. attack) for supervised models, or unlabelled data for anomaly detection.
  3. Detection: New traffic is fed to the model; if prediction ≠ “normal,” raise alert.
  • Example: A Random Forest trained on the UNSW-NB15 dataset classifies each TCP flow as normal or one of nine attack types with >90 % accuracy.

2. Application in OT Networks

  • Data Sources: Sensor readings (temperature, pressure), control commands (Modbus/DNP3), timing of operations.
  • Process:
  1. Baseline Profiling: Model typical sensor‐command relationships over time.
  2. Anomaly Scoring: Deviations from learned patterns (e.g. sudden pressure spike) score high and trigger alarms.
  • Example: An autoencoder trained on normal PLC sensor data flags reconstruction error when it encounters a replay attack.

3. Simple Models

  • Decision Tree:

    • How it works: Splits feature space by threshold tests (e.g. “if packet_count > 1000 then attack”).
    • Example: A tree built on KDD’99 features identifies DoS attacks by checking duration, src_bytes, and srv_count.

  • k-Nearest Neighbors (k-NN):

    • How it works: Computes distance (e.g. Euclidean) between a new sample and its k closest training points; classifies by majority vote.
    • Example: k-NN on flow features detects port scans because scan flows lie far from clusters of normal activity.

4. Advanced Models

  • Ensemble Methods (e.g., Random Forest, XGBoost):

    • How they work: Combine many decision trees to reduce overfitting and improve generalization.
    • Example: XGBoost on NSL-KDD data distinguishes complex mixtures of probes and U2R attacks with high precision.

  • Deep Learning (Autoencoders, LSTM):

    • Autoencoder: Learns compact representation of normal data; high reconstruction error indicates anomaly. Used on OT sensor streams to detect unusual process states.
    • LSTM (Recurrent NN): Learns temporal patterns in sequence data. Example: LSTM flagging abnormal sequences of Modbus commands in a SCADA network.

5. When to Use Which

  • Simple Models: Quick training, low resource use, easy to interpret. Good for smaller IT networks with clear feature separability.
  • Advanced Models: Handle high-dimensional or time-series data and complex attack patterns. Preferred in large-scale IT environments or critical OT systems where temporal context and subtle anomalies matter.
