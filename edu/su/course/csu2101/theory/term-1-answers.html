Section A. Very Short Answer Questions (2 marks each)

Table: Student(StudID, Name, MajorSubject, GPA)

1. Display students who opted “Mathematics”

SELECT StudID, Name, MajorSubject, GPA
FROM Student
WHERE MajorSubject = 'Mathematics';

2. Display all students in ascending order of GPA

SELECT StudID, Name, MajorSubject, GPA
FROM Student
ORDER BY GPA ASC;

3. Display students whose GPA > 8

SELECT StudID, Name, MajorSubject, GPA
FROM Student
WHERE GPA > 8;

4. Display students who did not opt “Computer Science”

SELECT StudID, Name, MajorSubject, GPA
FROM Student
WHERE MajorSubject <> 'Computer Science';

5. Display “Computer Science” students in descending GPA

SELECT StudID, Name, MajorSubject, GPA
FROM Student
WHERE MajorSubject = 'Computer Science'
ORDER BY GPA DESC;

Section B. Short Answer Questions (4 marks each)

Q1. Difference between DELETE and DROP

DELETE removes rows from a table but the table structure and its schema remain.

DELETE FROM Employee WHERE EmpID = 5;
  • Can be rolled back (if within a transaction).
  • Triggers fire; each row deletion is logged.

DROP completely removes the table (or other object) from the database.

DROP TABLE Employee;
  • Cannot be rolled back in most systems.
  • All data, indexes, constraints, and privileges on that table are lost.

Q2. GROUP BY vs ORDER BY

GROUP BY

  • Aggregates rows sharing the same values into summary rows.
  • Used with aggregate functions: SUM, COUNT, AVG, etc.
SELECT MajorSubject, AVG(GPA) AS AvgGPA
FROM Student
GROUP BY MajorSubject;

ORDER BY

  • Sorts the result‐set by one or more columns (ascending ASC or descending DESC).
  • Does not change the number of rows.
SELECT Name, GPA
FROM Student
ORDER BY GPA DESC;

Q3. WHERE vs HAVING

WHERE

  • Filters rows before any grouping or aggregation.
  • Cannot use aggregate functions in its condition.

HAVING

  • Filters groups after grouping/aggregation.
  • Can use aggregate functions.
SELECT MajorSubject, COUNT(*) AS Cnt
FROM Student
GROUP BY MajorSubject
HAVING COUNT(*) > 1;

Q4. Four common data constraints

  • PRIMARY KEY: Uniquely identifies each row; cannot be NULL.
  • FOREIGN KEY: Enforces referential integrity between two tables.
  • UNIQUE: Ensures all values in a column are distinct.
  • NOT NULL: Ensures a column cannot have NULL values.
  • CHECK: Enforces a Boolean condition on column values, e.g. CHECK (GPA BETWEEN 0 AND 10).

Section C. Long Answer Questions (8 marks each)

Q1. What is SQL? Discuss its types.

Definition: Structured Query Language (SQL) is the ANSI/ISO-standard declarative language for defining, querying, and manipulating data in relational database systems.

Purpose: Enables data definition, retrieval, modification, access control, and transaction management by specifying what operations to perform without procedural details.

Key Features:

  • Declarative syntax focuses on what to do, not how to do it.
  • Set-oriented operations process multiple rows in a single statement.
  • ACID-compliant transactions ensure atomicity, consistency, isolation, and durability.
  • Portable across major RDBMS platforms (Oracle, MySQL, SQL Server, PostgreSQL).
  • Extensible via stored procedures, functions, and triggers.

Main categories of SQL statements:

  • DDL (Data Definition Language): Defines or alters schema objects; commands: CREATE, ALTER, DROP, TRUNCATE.
  • DML (Data Manipulation Language): Inserts, updates, deletes, and merges data; commands: SELECT, INSERT, UPDATE, DELETE, MERGE.
  • DCL (Data Control Language): Manages user permissions; commands: GRANT, REVOKE.
  • TCL (Transaction Control Language): Controls transactions; commands: COMMIT, ROLLBACK, SAVEPOINT, SET TRANSACTION.
  • DQL (Data Query Language): Retrieves data using complex SELECT statements with filtering, grouping, and sorting.

Q2. What is Data Integrity (Data Constraints)? Discuss its types.

Definition: Data integrity comprises schema-level rules that enforce the accuracy, consistency, and reliability of data in a relational database.

Importance: Prevents invalid or contradictory data, upholds business rules, and guarantees reliable query and transaction results.

Types of Integrity Constraints:

  • Entity Integrity: Enforces a non-null primary key to uniquely identify each row; e.g., PRIMARY KEY (StudID).
  • Referential Integrity: Ensures foreign key values match existing primary key values or are null; e.g., FOREIGN KEY (DeptNo) REFERENCES Dept(DeptNo) ON DELETE CASCADE.
  • Domain Integrity: Restricts column values via data types, NOT NULL, UNIQUE, and CHECK constraints; e.g., CHECK (GPA BETWEEN 0.0 AND 10.0).
  • User-Defined Integrity: Implements complex business rules through triggers or stored procedures; e.g., a trigger rejecting OrderDate later than DeliveryDate.

Example:

CREATE TABLE Student (
  StudID INT PRIMARY KEY,
  Name   VARCHAR(50) NOT NULL,
  Email  VARCHAR(100) UNIQUE,
  GPA    DECIMAL(3,1) CHECK (GPA BETWEEN 0.0 AND 10.0),
  DeptNo INT,
  FOREIGN KEY (DeptNo) REFERENCES Dept(DeptNo) ON DELETE CASCADE
);

Conclusion: Applying these constraints ensures that database contents remain valid, consistent, and aligned with organizational policies.
