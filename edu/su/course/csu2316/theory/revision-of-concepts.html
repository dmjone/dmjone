DevOps

DevOps is a set of practices, principles, and tools that integrates software development (Dev) and IT operations (Ops) to deliver software faster, reliably, and continuously. It removes the traditional separation between development and operations by enabling collaboration, automation, and shared responsibility.

Why DevOps is Needed

Traditional software development follows a silo-based approach where developers focus on writing code while operations teams focus on stability. This creates the “Wall of Confusion”, leading to slow releases, frequent failures, and poor system reliability.

  • Problem: Slow deployments, manual errors, unstable systems
  • Solution: Automation + collaboration across the software lifecycle

Core Objectives of DevOps

  • Continuous Delivery: Release software frequently and safely
  • High Availability: Ensure systems remain accessible
  • Scalability: Handle increasing load automatically
  • Reliability: Minimize failures using automation
  • Collaboration: Shared ownership between Dev and Ops

DevOps Principles

  • Automation: Automate build, test, deployment, and infrastructure
  • Continuous Integration: Merge and test code frequently
  • Continuous Delivery: Always keep software deployable
  • Monitoring & Feedback: Detect issues early using metrics and logs
  • Infrastructure as Code (IaC): Manage infrastructure using versioned code

DevOps Lifecycle (Infinity Loop)

The DevOps lifecycle is represented as an infinite loop showing continuous improvement.

  • Plan: Requirement analysis and sprint planning
  • Code: Writing and managing source code using VCS
  • Build: Compile and package the application
  • Test: Automated unit, integration, and security testing
  • Release: Versioning and approval for deployment
  • Deploy: Automated deployment to production
  • Operate: Running and managing applications
  • Monitor: Logging, metrics, alerts, and feedback

DevOps Architecture (CI/CD Pipeline)

DevOps architecture is built around a CI/CD pipeline that automates software delivery.

  • Source Code Management: Git, GitHub
  • CI Server: Jenkins, GitHub Actions
  • Artifact Repository: Docker Registry, Nexus
  • Configuration Management: Ansible, Terraform
  • Containerization: Docker
  • Orchestration: Kubernetes
  • Monitoring: Prometheus, Grafana

Benefits of DevOps

  • Faster time-to-market
  • Reduced deployment failures
  • Improved system stability
  • Better resource utilization
  • Continuous improvement through feedback

DevOps vs Traditional Development

  • Traditional: Manual, slow, silo-based
  • DevOps: Automated, fast, collaborative

Real-World Enterprise Use

DevOps is widely used in enterprises to deploy microservices, cloud-native applications, and scalable systems where frequent updates and high availability are critical.

Git

Git is a Distributed Version Control System (DVCS) used to track changes in source code, manage versions, and enable collaboration among multiple developers. Each developer maintains a complete copy of the repository, including full history.

Why Git is Required

In software development, multiple developers work on the same codebase. Git prevents code loss, manages parallel development, and enables safe experimentation.

  • Tracks changes with complete history
  • Supports collaboration without overwriting work
  • Enables rollback to previous stable versions
  • Manages branching and merging

Git Architecture

Git follows a three-level architecture that separates working, staging, and committed code.

  • Working Directory: Actual project files
  • Staging Area (Index): Prepared snapshot for commit
  • Local Repository: Permanent history stored as commits
  • Remote Repository: Shared repository for collaboration

Git Workflow

Code flows through Git in a controlled manner:

Working Directory → Staging Area → Local Repository → Remote Repository

Git Objects

  • Blob: Stores file content
  • Tree: Stores directory structure
  • Commit: Snapshot with metadata
  • HEAD: Pointer to current branch

Branching in Git

A branch is an independent line of development that allows parallel work.

  • Master/Main: Stable production branch
  • Feature Branch: New feature development
  • Hotfix Branch: Emergency fixes

Merging in Git

  • Fast-forward Merge: Linear history
  • Three-way Merge: Uses common ancestor
  • Merge Conflict: Occurs when same code is modified

Common Git Commands

  • git init: Initialize repository
  • git clone: Copy remote repository
  • git status: Check file state
  • git add: Stage changes
  • git commit: Save snapshot
  • git log: View commit history
  • git branch: Manage branches
  • git checkout: Switch branches
  • git merge: Merge branches
  • git pull: Fetch + merge from remote
  • git push: Upload commits

Advantages of Git

  • Distributed and fault-tolerant
  • High performance
  • Strong data integrity (SHA-1 hashing)
  • Supports offline work

Git in DevOps

Git acts as the single source of truth in DevOps pipelines, triggering automated CI/CD workflows on every commit.

Git

Git is a Distributed Version Control System (DVCS) used to track changes in source code, manage versions, and support collaboration. Every developer has a complete copy of the repository along with its full history.

Git ensures that code changes are recorded as commits, allowing rollback, parallel development, and safe merging.

  • Working Directory: Actual project files
  • Staging Area: Selected changes prepared for commit
  • Local Repository: Stores committed snapshots
  • Remote Repository: Shared central repository

GitHub

GitHub is a cloud-based hosting platform for Git repositories that enables collaboration, code review, project management, and automation.

While Git is a tool, GitHub is a service that provides a web interface, access control, and collaboration features over Git repositories.

  • Repository: Project storage with code and history
  • Branch: Independent development line
  • Commit: Saved code snapshot
  • Pull Request (PR): Request to merge code changes
  • Issue: Bug or feature tracking
  • Fork: Personal copy of another repository

GitHub Workflow: Clone → Branch → Commit → Push → Pull Request → Review → Merge

GitHub Actions

GitHub Actions is a CI/CD automation service provided by GitHub to automatically build, test, and deploy code directly from a GitHub repository.

It works by triggering automated workflows in response to repository events.

  • Workflow: Automated process defined in YAML
  • Event: Trigger (push, pull_request)
  • Job: Set of steps executed on a runner
  • Step: Individual task or command
  • Runner: Machine that executes jobs

Workflow File Structure

Stored in .github/workflows/ directory.


name: CI Pipeline
on: [push]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - run: echo "Build Successful"

Role in DevOps: GitHub Actions enables Continuous Integration and Continuous Deployment by automating testing and deployment on every code change.

Git vs GitHub vs GitHub Actions

  • Git: Version control tool
  • GitHub: Repository hosting and collaboration platform
  • GitHub Actions: Automation and CI/CD service

Amazon EC2 (Elastic Compute Cloud)

Amazon EC2 is a cloud service that provides resizable virtual servers (instances) to run applications on the AWS cloud. It allows users to provision computing resources on demand instead of maintaining physical servers.

Working Principle: User selects an instance type → chooses AMI → launches instance → application runs on virtual hardware.

  • Instance: Virtual server
  • AMI (Amazon Machine Image): OS + preconfigured software
  • Instance Types: Compute, Memory, Storage optimized
  • Key Pair: Secure SSH access
  • Security Group: Virtual firewall

Use Cases: Web hosting, application servers, testing environments, backend processing.

Amazon S3 (Simple Storage Service)

Amazon S3 is an object storage service designed to store and retrieve unlimited data with high durability and availability.

Data in S3 is stored as objects inside buckets. Each object consists of data, metadata, and a unique key.

  • Bucket: Logical container for objects
  • Object: File stored in S3
  • Key: Unique identifier for object
  • Storage Classes: Standard, IA, Glacier

Features: 99.999999999% durability, versioning, lifecycle management, encryption.

Use Cases: Backup, static website hosting, media storage, data lakes.

IAM (Identity and Access Management)

IAM is a security service that controls who can access AWS resources and what actions they can perform.

IAM follows the principle of least privilege, granting only required permissions.

  • User: Individual identity
  • Group: Collection of users
  • Role: Temporary permission set
  • Policy: JSON document defining permissions

Policy Evaluation: Explicit Deny > Explicit Allow > Default Deny

Use Cases: Secure access control, service-to-service authentication, compliance.

Relationship Between EC2, S3 and IAM

  • EC2: Computes and runs applications
  • S3: Stores application data
  • IAM: Secures access between users and services

Example: An EC2 instance accesses an S3 bucket securely using an IAM Role.

Docker

Docker is a containerization platform used to package an application along with its dependencies into a lightweight, portable container that runs consistently across different environments.

Docker eliminates the problem of “it works on my machine” by ensuring environment consistency.

Why Docker is Used

  • Provides environment consistency
  • Faster application deployment
  • Efficient resource utilization
  • Supports microservices architecture

Docker Architecture

Docker follows a client–server architecture.

  • Docker Client: CLI used by user
  • Docker Engine: Core service managing containers
  • Docker Daemon: Background service handling requests
  • Docker Registry: Stores Docker images (Docker Hub)

Flow: Client → Engine → Registry

Key Docker Components

  • Image: Read-only template for containers
  • Container: Running instance of an image
  • Dockerfile: Instructions to build image
  • Docker Hub: Public image repository

Dockerfile Instructions

  • FROM: Base image
  • RUN: Execute commands during build
  • COPY: Copy files to image
  • WORKDIR: Set working directory
  • CMD: Default command
  • ENTRYPOINT: Fixed executable
  • EXPOSE: Document port
  • ENV: Environment variables

CMD vs ENTRYPOINT

  • CMD: Default, overridable command
  • ENTRYPOINT: Fixed command, CMD acts as argument

Docker Workflow

Dockerfile → Image → Container

  • Build image using Dockerfile
  • Store image in registry
  • Run container from image

Common Docker Commands

  • docker build: Build image
  • docker run: Run container
  • docker ps: List running containers
  • docker images: List images
  • docker stop: Stop container
  • docker rm: Remove container

Docker vs Virtual Machine

  • Docker: Lightweight, shared OS kernel
  • VM: Heavyweight, full OS per VM

Docker in DevOps

Docker enables fast, reliable deployments and forms the foundation of CI/CD pipelines and Kubernetes orchestration.

Kubernetes (K8s)

Kubernetes is an open-source container orchestration platform used to automate the deployment, scaling, management, and healing of containerized applications.

It manages multiple containers running across multiple machines as a single cluster.

Why Kubernetes is Needed

Docker manages single containers well, but fails at large-scale container management. Kubernetes solves this by providing orchestration.

  • Automatic container scheduling
  • Self-healing (restart, replace)
  • Auto-scaling of applications
  • Load balancing and service discovery

Kubernetes Architecture

Kubernetes follows a master–worker (control plane–node) architecture.

Control Plane Components
  • API Server: Entry point for all requests
  • etcd: Distributed key-value store for cluster state
  • Scheduler: Assigns pods to nodes
  • Controller Manager: Ensures desired state
2.2 Worker Node Components
  • Kubelet: Communicates with control plane
  • Container Runtime: Runs containers (Docker, containerd)
  • Kube-proxy: Network routing and load balancing

Kubernetes Objects

  • Pod: Smallest deployable unit
  • Node: Physical or virtual machine
  • Deployment: Manages pod replicas
  • ReplicaSet: Maintains desired pod count
  • Service: Exposes pods
  • Namespace: Logical isolation
  • ConfigMap: Non-sensitive configuration
  • Secret: Sensitive data

Pod and Deployment

  • Pod: One or more containers sharing network and storage
  • Deployment: Handles rolling updates and rollbacks

Kubernetes Services

  • ClusterIP: Internal access only
  • NodePort: Exposes service on node IP
  • LoadBalancer: Cloud provider load balancer

Kubernetes Workflow

YAML → API Server → Scheduler → Node → Pod

The user defines the desired state, Kubernetes ensures the actual state matches it.

Kubernetes Manifest File

A manifest file is a YAML file used to define Kubernetes objects.

  • apiVersion: API version
  • kind: Object type
  • metadata: Name, labels
  • spec: Desired state

Common kubectl Commands

  • kubectl apply: Create or update resource
  • kubectl get: List resources
  • kubectl describe: Detailed information
  • kubectl delete: Remove resource
  • kubectl logs: View pod logs
  • kubectl exec: Execute inside pod

Kubernetes in DevOps

Kubernetes is the backbone of cloud-native and microservices-based DevOps systems, enabling scalability, reliability, and automation.

End-to-End DevOps “Hello World” Workflow

This workflow demonstrates a complete DevOps pipeline starting from local development to automated deployment using Git, GitHub, GitHub Actions, Docker, Kubernetes, and EC2. Each step explains what happens and how it works internally.

Project Folder Creation

A project folder represents the application workspace where source code and configuration files reside.


mkdir hello-devops
cd hello-devops

This folder becomes the root of the project and later the Git repository.

Initialize Git Repository

Git initialization creates a .git directory that stores version history and metadata.


git init

Git now starts tracking changes inside this folder.

Create Hello World Application

A minimal application is created to demonstrate deployment.


echo "Hello DevOps World" > index.html

This file represents application output.

Commit Code to Git

Files are moved through Git’s workflow:

Working Directory → Staging Area → Local Repository


git add .
git commit -m "Initial hello world app"

The commit creates an immutable snapshot of the project.

Push Code to GitHub

GitHub acts as a remote repository for collaboration and automation.


git branch -M main
git remote add origin https://github.com/user/hello-devops.git
git push -u origin main

The local repository is now synchronized with GitHub.

Create a Feature Branch

Branches allow parallel development without affecting stable code.


git checkout -b feature-update

This creates an isolated line of development.

Edit Code in Feature Branch

A change is made independently.


echo "Hello from Feature Branch" >> index.html
git add .
git commit -m "Updated hello message"

The change is recorded only in the feature branch.

Merge Feature Branch to Main

Merging integrates changes into the stable branch.


git checkout main
git merge feature-update
git push origin main

Git performs a three-way merge and updates GitHub.

Connect to EC2 Instance

EC2 provides the compute infrastructure for deployment.


ssh -i key.pem ec2-user@EC2_PUBLIC_IP

Secure Shell (SSH) allows remote server access.

Install Docker and Kubernetes on EC2

Docker is required to run containers, Kubernetes to orchestrate them.


sudo yum install docker -y
sudo systemctl start docker
sudo yum install kubectl -y

Docker Engine runs containers, kubectl manages the cluster.

GitHub Actions Workflow Creation

GitHub Actions automates tasks on code push events.


name: DevOps Pipeline
on: [push]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - run: docker build -t hello-devops .

This workflow triggers automatically on every push.

Docker Image Creation

Dockerfile defines how the application is packaged.


FROM nginx
COPY index.html /usr/share/nginx/html

Docker builds an image containing the application and web server.

Kubernetes Cluster Creation

Kubernetes manages containers as pods inside a cluster.


kubectl create deployment hello-devops --image=hello-devops
kubectl expose deployment hello-devops --type=NodePort --port=80

Kubernetes ensures desired state, scalability, and self-healing.

End-to-End Flow Summary

  • Developer writes code locally
  • Git tracks and versions changes
  • GitHub hosts code and triggers automation
  • GitHub Actions builds Docker image
  • Docker packages application
  • Kubernetes deploys and manages containers
  • EC2 provides compute infrastructure
