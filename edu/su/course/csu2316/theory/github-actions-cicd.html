4. GitHub Actions

GitHub Actions is a native automation engine inside GitHub that executes workflows in response to events occurring in a repository. Conceptually, it is the “engine room” that converts source code changes into verified, deployable software without human intervention.

4.1 Definition

GitHub Actions is a CI/CD and automation service where workflows (written in YAML) define what should run, when it should run, and where it should run. Each workflow is triggered by repository events and executed on runners.

4.2 Why GitHub Actions Is Used

  • Automates build & test: Every code change is validated automatically.
  • Enables CI/CD: Continuous Integration (CI) and Continuous Deployment (CD) pipelines are defined as code.
  • Reduces manual work: No human-triggered builds or deployments.
  • Improves reliability: Same steps run the same way every time.
  • Tight GitHub integration: No external CI tool required.
How does GitHub Actions eliminate the "it works on my machine" problem?

A: By guaranteeing that builds and tests execute in a clean, consistent runner environment every single time, removing local configuration drift.

4.3 Where GitHub Actions Is Used

  • Open-source projects for automated testing on pull requests
  • Enterprise applications for CI/CD pipelines
  • DevOps automation (linting, security scans, releases)
  • Non-CI tasks: issue labeling, notifications, scheduled jobs

4.4 How GitHub Actions Is Used (Mental Model)

Think of GitHub Actions as an event → pipeline → execution system.

  • An event happens (e.g., push).
  • A workflow is triggered.
  • The workflow starts one or more jobs.
  • Each job runs multiple steps on a runner.

4.5 Key Concepts

Concept Meaning
Workflow Complete automation pipeline defined in YAML
Event Trigger that starts a workflow
Job Group of steps executed on the same runner
Step Single command or action
Action Reusable automation unit
Runner Machine that executes jobs
What is the relationship between a Job and a Runner?

A: A Job is a set of steps that execute on a single Runner. Each job gets a fresh runner instance, ensuring isolation.

4.6 Runners and Their Types

A runner is the execution environment for jobs.

  • GitHub-hosted runners: Managed by GitHub (Ubuntu, Windows, macOS).
  • Self-hosted runners: Your own machines (on-prem or cloud).

Runner selection is defined using:

runs-on: ubuntu-latest

4.7 Workflow File

Workflows are written in YAML and stored inside:

.github/workflows/

Each file represents one workflow.

name: CI Pipeline

on: push

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm install
      - run: npm test

4.8 Events (Triggers)

Events define when workflows run.

4.8.1 push
on:
  push:
    branches: [ "main" ]
4.8.2 pull_request
on:
  pull_request:
    branches: [ "main" ]
4.8.3 workflow_dispatch (Manual)
on:
  workflow_dispatch:
4.8.4 schedule (Cron-based)
on:
  schedule:
    - cron: "0 0 * * *"

Runs daily at midnight UTC.

Can a workflow run on a schedule instead of a code change?

A: Yes, using the schedule event with cron syntax (e.g., for nightly builds or periodic cleanups).

4.9 Jobs and Steps: Execution Hierarchy

Hierarchy: Workflow → Job → Step

  • Jobs run in parallel by default.
  • Steps run sequentially inside a job.
  • Jobs can depend on each other using needs.

4.10 Actions

Actions are reusable building blocks.

  • Official actions: Maintained by GitHub
  • Community actions: Published by developers
  • Custom actions: Created by you
- uses: actions/setup-node@v4
  with:
    node-version: '18'

4.11 Environment Variables & Secrets

Secrets store sensitive data securely.

env:
  NODE_ENV: production
- run: echo "${{ secrets.AWS_KEY }}"

Secrets are encrypted and never logged.

4.13 Matrix Strategy (Parallel Testing at Scale)

A matrix allows the same job to run across multiple environments (OS, language versions). This is critical in real-world CI to detect environment-specific failures.

jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        node: [16, 18, 20]
    steps:
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
      - run: npm test

4.14 Caching Dependencies (Performance Optimization)

Caching avoids downloading dependencies on every run, drastically reducing pipeline execution time.

- uses: actions/cache@v4
  with:
    path: ~/.npm
    key: npm-${{ runner.os }}-${{ hashFiles('package-lock.json') }}

4.15 Artifacts (Persisting Build Outputs)

Artifacts allow sharing files (reports, binaries) between jobs or storing them after workflow completion.

Why do files created in one job disappear in the next job?

A: Because each job runs in a fresh, isolated environment. You must use Artifacts to persist data between jobs. See why isolation matters.

- uses: actions/upload-artifact@v4
  with:
    name: test-report
    path: reports/

4.16 Job Dependencies & Failure Control

Jobs can be conditionally executed based on success, failure, or always-run semantics.

- run: npm run cleanup
  if: always()

Common conditions: success(), failure(), always()

4.17 Permissions & Security Model

GitHub Actions follows the principle of least privilege using fine-grained permissions.

permissions:
  contents: read
  deployments: write

This prevents workflows from over-accessing repository resources.

4.18 Environments & Deployment Protection

Environments represent deployment targets (dev, staging, prod) and support approval gates.

environment:
  name: production

Used to enforce manual approvals and environment-specific secrets.

4.19 Workflow Reusability

Reusable workflows reduce duplication across repositories and teams.

uses: org/repo/.github/workflows/ci.yml@main

4.20 Common Failure Patterns (Operational Insight)

  • Missing secrets → runtime failure
  • Wrong runner OS → dependency mismatch
  • Uncached dependencies → slow pipelines
  • No job isolation → hidden coupling

4.21 Workflow Execution Context

Each job runs in a fresh, isolated environment. No filesystem state is shared across jobs unless explicitly passed via artifacts or caches.

Why must you checkout code explicitly using actions/checkout?

A: Because the runner starts empty. It does not contain your repository code until you explicitly instruct the workflow to fetch it.

4.22 Checkout Depth & Repository Context

By default, only the latest commit is fetched. Full history is not available unless specified.

- uses: actions/checkout@v4
  with:
    fetch-depth: 0

Required for versioning, changelogs, and Git history analysis.

4.23 Expressions & Context Objects

GitHub Actions uses expression syntax for dynamic behavior.

General form: ${{ context.property }}

  • github: repo, branch, actor
  • env: environment variables
  • secrets: encrypted values
  • matrix: matrix values
if: github.ref == 'refs/heads/main'

4.24 Timeouts & Resource Limits

Jobs have execution limits to prevent runaway pipelines.

timeout-minutes: 15

Exam-relevant for cost, reliability, and failure control.

4.25 GitHub Actions vs Jenkins

  • Actions: YAML + GitHub-native
  • Jenkins: Server-based, plugin-heavy
  • Actions: Event-driven
  • Jenkins: Pipeline-triggered

4.26 Simple Workflow Example

Here is the absolute simplest way to see GitHub Actions in action. This "Hello World" workflow prints a message to the console.

1. Create the File

Inside your repository, create this file path:

.github/workflows/hello.yml

2. Add the Content

name: Hello World

on: [push]

jobs:
  say-hello:
    runs-on: ubuntu-latest
    steps:
      - run: echo "Hello, GitHub Actions!"

3. How it Works

  • on: [push] → Runs automatically whenever you push code.
  • runs-on: ubuntu-latest → GitHub provides a fresh Linux server.
  • run: echo ... → The server executes this shell command.
