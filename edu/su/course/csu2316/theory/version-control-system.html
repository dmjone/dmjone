Version Control System (VCS)

2025, Dec 16

1. Version Control System (VCS)

A Version Control System (VCS) is a system that records how a set of files changes over time so that any past state can be reconstructed exactly. Conceptually, VCS treats a project as a sequence of well-defined states instead of a single evolving folder.

1.1 Why VCS is Required

Software development is change-intensive. Without VCS, change becomes destructive instead of constructive.

  • Change is inevitable: Requirements evolve, bugs are fixed, features are added.
  • Human memory is unreliable: VCS becomes the system’s memory.
  • Collaboration causes conflicts: Multiple people modify the same files.
  • Failures must be reversible: Mistakes should be undoable, not fatal.

Without VCS, teams rely on file copies like final_v3_really_final.zip, which destroys traceability and correctness.

Why is relying on manual backups like "final_v2.zip" dangerous?

A: It lacks traceability (who changed what), reversibility (undoing specific mistakes), and concurrency (merging collaborative work).

1.2 Where VCS Is Used

  • Software Development: Source code, configuration, scripts.
  • DevOps & Infrastructure: IaC files (Terraform, YAML, Helm).
  • Documentation: Manuals, policies, research papers.
  • Data & ML: Experiment tracking, model versions.

Any domain where correctness over time matters benefits from VCS.

1.3 How a VCS Works

Think of a project as a timeline of snapshots.

  • A snapshot captures the exact state of tracked files at a moment.
  • Each snapshot is immutable and identifiable.
  • New changes create new snapshots instead of overwriting old ones.

Formally, a VCS maintains a directed history of states, not just files.

1.4 Core Functions of a VCS

  • Track Changes: What changed, when, by whom, and why.
  • Collaboration: Parallel work without overwriting others.
  • Revert: Move backward to any previous stable state.
  • Branch: Create independent lines of development.
  • Merge: Combine independent lines safely.
  • History: Maintain an auditable evolution of the project.

1.5 Types of Version Control Systems

1.5.1 Local Version Control System

Stores versions only on a single machine.

  • Example: RCS
  • Limitation: No collaboration, single point of failure.
1.5.2 Centralized Version Control System (CVCS)

A central server stores the repository; users pull working copies.

  • Examples: SVN, Perforce
  • Pros: Central authority, simpler model.
  • Cons: Server failure blocks all work.
1.5.3 Distributed Version Control System (DVCS)

Every user has a full copy of the repository including history.

  • Examples: Git, Mercurial
  • Pros: Offline work, resilience, flexible workflows.
  • Cons: Conceptually richer, steeper learning curve.
What is the main resilience advantage of specific Distributed VCS (DVCS) like Git?

A: Every user has a full copy of the repository history. If the central server fails, any client repository can restore it.

1.6 Repository (Repo)

A repository is a managed database of project states.

  • Tracked files: Files under version control.
  • History: Ordered sequence of commits.
  • Metadata: Author, timestamps, messages.
  • References: Branches, tags.

A repo is not just storage; it is the authoritative history of truth.

1.7 File States in a Version Control System

1.7.1 Untracked

Files exist in the working directory but are unknown to VCS.

1.7.2 Unmodified

Tracked files identical to the last committed snapshot.

1.7.3 Modified

Tracked files changed but not yet selected for commit.

1.7.4 Staged

Modified files explicitly marked to be part of the next snapshot.

What is the purpose of the "Staged" state (Index)?

A: It allows selective commits. You can choose exactly which modified files go into the next snapshot. Learn more about the Index in Git.

1.7.5 Committed

Files safely stored as an immutable snapshot in repository history.

1.7.6 Unmerged

Conflicting changes from different branches requiring human resolution.

1.7.7 Merged

Conflicts resolved and combined into a single consistent state.

1.8 Branching Concept

A branch represents an independent line of evolution.

  • Main branch: stable, production-ready history.
  • Feature branches: isolated experimentation.

Branches allow safe parallelism without destabilizing the main codebase.

1.9 Merge Concept

Merging reconciles divergent histories. Merging formalizes collaboration mathematically, not manually.

  • Fast-forward merge: Linear history extension.
  • Three-way merge: Combines changes using a common ancestor.

1.10 Commit as a Fundamental Unit

A commit is an atomic, immutable snapshot. Commits form a directed acyclic graph (DAG) of project evolution.

  • Has a unique identifier (hash).
  • References its parent commit(s).
  • Contains a message explaining intent.

1.11 VCS in DevOps Context

VCS becomes the single source of truth. In DevOps, version control governs not only code, but systems.

  • CI pipelines trigger on commits.
  • CD deploys specific commit hashes.
  • Rollback equals redeploying an earlier commit.
How does a VCS commit trigger a DevOps pipeline?

A: The commit acts as an event signal. CI systems detect the new hash and automatically start building that specific version. See how CI fits into the lifecycle.

1.12 VCS Workflow Models

Workflow defines how teams use branches and merges in practice. This is where theory meets real projects.

  • Centralized Workflow: One main branch, direct commits (used in small teams).
  • Feature Branch Workflow: Each feature in its own branch; merged after review.
  • Gitflow Workflow: Structured branches (main, develop, feature, release, hotfix).
  • Trunk-Based Development: Short-lived branches, frequent merges to main.

1.13 Merge Conflicts

A merge conflict occurs when two branches change the same part of a file differently. Conflicts are not errors; they are signals of parallel work.

  • VCS cannot decide intent automatically.
  • Human resolves semantic meaning.
  • Conflict markers explicitly show competing changes.

1.14 Tags and Releases

Tags mark important commits such as versions or releases. Tags convert continuous history into named milestones.

  • Lightweight tags: Simple pointers.
  • Annotated tags: Metadata-rich, signed references.

1.15 Remote Repositories

A remote repository is a shared reference point for collaboration. Push = publish history, Pull = integrate external history.

  • Local repo: personal workspace.
  • Remote repo: synchronization hub.

1.16 Ignore Rules (concept of .gitignore)

Not all files should be versioned. Ignore rules enforce clean, portable repositories.

  • Build artifacts
  • Secrets and credentials
  • Temporary files

1.17 Revert vs Reset

This distinction is essential for collaborative safety.

  • Revert: Creates a new commit that undoes a change (history-safe).
  • Reset: Moves branch pointer backward (history-rewriting).

1.18 VCS as a Directed Graph

Repository history is a Directed Acyclic Graph (DAG). This mental model explains branching, merging, and rebasing precisely.

  • Commits are nodes.
  • Parent links define direction.
  • Branches are movable references.

In the "Directed Acyclic Graph" (DAG) model, what do the nodes represent?

A: Nodes represent commits (snapshots), while edges represent the parent relationship pointing backward in time.

1.19 Security and Accountability in VCS

  • Commit authorship enables accountability.
  • Signed commits ensure integrity.
  • Access control protects history.

VCS is also a security control system, not just a developer tool.

1.20 Common Misconceptions

  • VCS ≠ Backup system
  • Git ≠ GitHub
  • Branch ≠ Copy of code

Correcting misconceptions prevents architectural misuse.
