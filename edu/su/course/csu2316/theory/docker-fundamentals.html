8. Docker

Docker is an OS-level virtualization platform that packages an application and everything it needs (libraries, runtime, configs) into a single portable unit called a container. Think of Docker as a standardized shipping container for software: once packed, it runs the same everywhere.

8.1 Where Docker Is Used

Docker is used wherever consistency, speed, and portability matter.

  • Development: Same environment for all developers (no “works on my machine”).
  • Testing: Spin up identical test environments on demand.
  • CI/CD pipelines: Build, test, and package applications automatically.
  • Microservices: Each service runs in its own isolated container.
  • Cloud & On-Prem: Same container runs on laptop, data center, or cloud.
Does Docker solve the "It works on my machine" problem?

A: Yes, because it packages the exact same environment (libraries, runtime, configs) that runs identically on development, testing, and production servers.

8.2 Why Docker Is Used

Docker solves environment inconsistency and deployment friction.

  • Consistency: Same app behavior across dev, test, prod.
  • Speed: Containers start in seconds (no full OS boot).
  • Resource efficiency: Shares host OS kernel.
  • Portability: Build once, run anywhere.
  • Scalability: Easy horizontal scaling.

8.3 Core Components

  • Docker Engine: Core runtime that builds and runs containers.
  • Docker Image: Read-only template with app + dependencies.
  • Docker Container: Running instance of an image.
  • Dockerfile: Text file defining how to build an image.
  • Docker Hub: Cloud registry for storing and sharing images.

Docker follows a client–server model.

  • Client: CLI (`docker` command).
  • Engine (Daemon): Builds images, runs containers.
  • Registry: Stores images (Docker Hub or private).

Flow: Client → Engine → Registry → Engine → Container

8.5 Dockerfile

A Dockerfile is a reproducible blueprint for creating images. Each instruction creates a new image layer.

8.5.1 Common Dockerfile Instructions
Instruction Purpose
FROM Base image
RUN Execute commands during build
COPY Copy files from host
ADD Copy + extract archives / URLs
WORKDIR Set working directory
CMD Default command
ENTRYPOINT Fixed executable
EXPOSE Document port usage
ENV Set environment variables
USER Run container as specific user
What is the purpose of the FROM instruction in a Dockerfile?

A: It specifies the Base Image (e.g., FROM python:3.9) that your container starts with. It is essentially the "parent" of your image.

8.5.2 CMD vs ENTRYPOINT

CMD provides a default command that can be overridden.

ENTRYPOINT defines a fixed executable.

Aspect CMD ENTRYPOINT
Nature Default Fixed
Override Easy Difficult
Use case Flexible behavior Executable containers 
ENTRYPOINT ["python"]
CMD ["app.py"]
When should you use ENTRYPOINT instead of CMD?

A: When you want your container to run as a fixed executable that receives arguments (e.g., a CLI tool), rather than a general-purpose shell.

8.5.3 COPY vs ADD

COPY is simple and predictable.

ADD adds extra features.

Aspect COPY ADD
Function File copy Copy + extract + URL
Best practice Preferred Use only if needed
8.5.4 Example: Node.js Hello World Workflow

Let's create a simple Node.js application, containerize it, and run it. Follow these steps:

Step 1: Create the Application

Create a file named app.js with a simple HTTP server:

const http = require('http');

const hostname = '0.0.0.0';
const port = 3000;

const server = http.createServer((req, res) => {
  res.statusCode = 200;
  res.setHeader('Content-Type', 'text/plain');
  res.end('Hello form Docker Container!\n');
});

server.listen(port, hostname, () => {
  console.log(`Server running at http://${hostname}:${port}/`);
});
Step 2: Create the Dockerfile

Create a file named Dockerfile (no extension) in the same directory:

# Use the official Node.js image.
FROM node:14

# Set the working directory inside the container.
WORKDIR /usr/src/app

# Copy the rest of the application code.
COPY . .

# Expose the port the app runs on.
EXPOSE 3000

# Define the command to run the app.
CMD [ "node", "app.js" ]
Step 3: Build the Image

Run the build command in your terminal. We tag it as node-hello. The . tells Docker to look for the Dockerfile in the current directory.

docker build -t node-hello .
Step 4: Run the Container

Run the container, mapping host port 3000 to container port 3000.

docker run -p 3000:3000 node-hello
Step 5: Interact

Open your browser to http://localhost:3000 or use curl:

curl localhost:3000

You should see: Hello from Docker Container!

8.6 Common Docker Commands

  • docker build: Build image from Dockerfile.
  • docker run: Create and start a container.
  • docker ps: List running containers.
  • docker images: List images.
  • docker stop: Stop a running container.
docker build -t myapp .
docker run -d -p 8080:80 myapp
docker ps
docker images
docker stop container_id
Which command lists only the running containers?

A: docker ps. use docker ps -a to see stopped containers as well.

8.7 Docker Hub

Docker Hub is a centralized cloud registry.

  • Public & Private images
  • Official images (nginx, node, python)
  • Versioned tags
  • Easy sharing & reuse

8.8 Enterprise-Relevant Additions

  • Image Layers & Caching: Faster rebuilds.
  • Volumes: Persist data outside containers.
  • Networking: Bridge, host, overlay.
  • Security: Non-root USER, minimal base images.
  • Docker vs Kubernetes: Docker builds containers; Kubernetes orchestrates them.

8.9 Container Lifecycle

Understanding the container lifecycle is critical for operating Docker in real systems.

  • Created: Container object exists but not running.
  • Running: Process inside container is executing.
  • Paused: Execution temporarily suspended.
  • Stopped: Process exited, container still exists.
  • Removed: Container metadata deleted.

Real-world mapping: CI creates → CD runs → Ops stops/removes.

8.10 Docker Volumes vs Bind Mounts

State management is essential in production.

Aspect Volume Bind Mount
Managed by Docker Yes No
Portability High Low
Production use Preferred Mainly dev 
docker volume create data_vol
docker run -v data_vol:/app/data myapp
Why is using a Volume better than saving data inside the container layer?

A: Volumes are persistent (survive container deletion) and offer better performance. Data written to the container's writable layer is lost when the container is removed.

8.11 Docker Networking Basics

Containers communicate using virtual networks.

  • Bridge: Default, container-to-container on same host.
  • Host: No network isolation (performance use-case).
  • Overlay: Multi-host communication (Swarm/K8s).

Enterprise insight: Microservices rely on bridge/overlay networking.

8.12 Image Tagging & Versioning

Tags are deployment contracts.

  • latest: Unsafe for production.
  • Semantic tags: v1.0, v1.1
  • Immutable images: New build = new tag.
docker tag myapp:1.0 myrepo/myapp:1.0
docker push myrepo/myapp:1.0

8.13 Docker Security Essentials

Security failures usually come from defaults.

  • Run as non-root using USER.
  • Minimal base images (alpine, distroless).
  • No secrets in images.
  • Read-only containers when possible.

8.14 Docker vs VM

This distinction matters in architecture decisions.

  • VM: Infrastructure isolation.
  • Docker: Application isolation.
  • Reality: Docker runs inside VMs in cloud.

8.15 Docker in DevOps Pipeline

Docker is not a replacement for CI/CD — it is a building block.

  • CI builds Docker image.
  • Image pushed to registry.
  • CD deploys container.
  • Orchestrator manages runtime.

This directly connects Docker → Kubernetes → Cloud.
