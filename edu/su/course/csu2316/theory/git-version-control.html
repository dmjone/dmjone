2. Git

Git is a Distributed Version Control System (DVCS) designed to manage source code changes efficiently, safely, and collaboratively. It records the complete history of a project and allows multiple developers to work in parallel without conflicts.

2.1 Definition & Distributed Nature

In Git, every developer has a full copy of the repository, including complete history. There is no single point of failure.

  • Distributed VCS: Each clone contains the entire project history.
  • Offline work: Commits, diffs, logs work without internet.
  • Peer-to-peer: Any repository can act as a source or destination.
Why is Git considered "Distributed"?

A: Because every client has a full copy of the repository and history, distinct from the central server, allowing peer-to-peer syncing and offline work.

2.2 What Git Does & Why We Use It

Git converts file changes into a timeline of snapshots.

  • Tracks change: Every commit is a permanent snapshot.
  • Enables collaboration: Multiple developers modify safely.
  • Supports branching: Isolated parallel development.
  • Ensures rollback: Any previous state can be restored.
  • Integrity: Content addressed using cryptographic hashes.

Real-world analogy: Git is like a time machine + shared notebook for code.

2.3 Git Architecture (Internal Mental Model)

Git is not file-based; it is snapshot-based. Every commit points to a tree of file states.

  • Blob: File content
  • Tree: Directory structure
  • Commit: Snapshot + metadata
  • HEAD: Pointer to current commit

Everything is referenced using a hash (SHA-1 / SHA-256).

2.4 Core Components

2.4.1 Repository

The repository is the database of history. Stored inside .git/.

  • Commits
  • Branches
  • Tags
  • Configuration
2.4.2 Working Tree (Working Directory)

This is the actual project files you edit.

Files can be:

  • Untracked
  • Modified
  • Unmodified
2.4.3 Staging Area (Index)

The staging area is a pre-commit buffer.

It defines what exactly goes into the next commit.

Think of it as a review table before saving history.

Why use a Staging Area instead of committing directly?

A: It acts as a "review table" or buffer, letting you curate exact changes for a clean commit history rather than dumping everything in.

2.5 Git Workflow (End-to-End)

Working Directory → Staging Area → Local Repository → Remote Repository

  • Edit: Modify files in working directory
  • Stage: Select changes using git add
  • Commit: Create snapshot in local repo
  • Push: Sync commits to remote
  • Pull: Fetch + merge from remote

2.6 Daily Use Git Commands (Conceptual + Practical)

2.6.1 Repository Setup & Info

git init              # Create a new repository
git clone URL         # Copy a remote repository
git status            # Show file states
git log               # View commit history
git log --oneline     # Compact history
2.6.2 Staging & Committing

git add file          # Stage a file
git add .             # Stage all changes
git commit -m "msg"   # Create commit
git commit --amend    # Modify last commit
2.6.3 Inspecting Changes

git diff              # Working vs staging
git diff --staged     # Staging vs last commit
git show              # Show commit details
2.6.4 Branching & Navigation

git branch            # List branches
git branch name       # Create branch
git checkout name     # Switch branch
git checkout -b name  # Create + switch
git merge branch      # Merge branch
2.6.5 Remote Operations

git remote add origin URL   # Add remote
git remote -v               # View remotes
git push origin main        # Push commits
git pull                    # Fetch + merge
git fetch                   # Download updates
2.6.6 File Operations

git rm file            # Remove file
git mv old new         # Rename file
git restore file       # Discard changes
git reset file         # Unstage file
2.6.7 Advanced Daily Commands

git stash              # Temporarily save changes
git stash pop          # Restore stashed changes
git blame file         # Who changed what line
git reflog             # History of HEAD
git clean -f           # Remove untracked files
When would you use git stash?

A: When you need to switch contexts (e.g., fix a bug on another branch) but aren't ready to commit your current "dirty" work.

2.7 Conceptual Summary

Git is:

  • A graph of snapshots
  • A local-first system
  • A content-addressed database
  • A collaboration backbone of DevOps

If you understand: working tree → index → commit → remote, you understand Git.

2.8 HEAD & References

HEAD is a symbolic pointer to the current commit you are working on.

  • Normally points to a branch (e.g., main)
  • Detached HEAD → points directly to a commit
  • Every checkout moves HEAD

Understanding HEAD explains checkout, reset, and revert behavior.

2.9 Branching Strategy

A branch is just a movable pointer, not a copy of files.

  • main: stable production-ready code
  • feature/*: isolated development
  • hotfix/*: emergency production fixes

Git encourages short-lived branches + frequent merges.

2.10 Merge Conflicts

A merge conflict occurs when Git cannot decide which change is correct.

  • Same line edited differently in two branches
  • Git pauses and asks the developer to decide

Conflict markers:


<<<<<< HEAD
your change
=======
incoming change
>>>>>> branch

Conflicts are design-time decisions, not errors.

2.11 git pull vs git fetch (Critical Distinction)

git pull = git fetch + git merge

  • fetch: download history safely
  • merge: modify local branch

Professional workflow prefers fetch before manual merge.

Why is git fetch safer than git pull?

A: fetch only downloads changes for review; pull downloads AND automatically merges, which can surprise you with conflicts.

2.12 reset vs revert (History Safety)
  • git reset: rewrites history (local use)
  • git revert: adds inverse commit (safe for shared repos)

Rule: Never rewrite public history.

2.13 .gitignore (Signal vs Noise Control)

.gitignore defines files Git must never track.

  • Build artifacts
  • Secrets
  • Local configs

node_modules/
.env
*.log

This preserves repository purity.

2.14 Tags (Release Markers)

Tags mark important commits (versions/releases).

  • Immutable pointers
  • Used for deployment & rollback

git tag v1.0
git push origin v1.0

2.15 Industry Readiness Check

If a student can confidently explain:

  • HEAD movement
  • Index purpose
  • fetch vs pull
  • reset vs revert
  • merge conflict resolution

They are Git-competent for DevOps, backend, and collaborative systems.
