7. Amazon EC2 (Elastic Compute Cloud)

Amazon EC2 is a cloud service that provides virtual servers (compute power) on demand. Instead of buying physical servers, organizations rent compute capacity and control it fully—OS, software, networking, and storage. EC2 is the foundational compute layer of AWS and the default execution environment for most enterprise workloads.

7.1 What EC2 Is

Think of EC2 as a remote computer in AWS data centers that you can create, resize, start, stop, or destroy programmatically. Each EC2 instance behaves like a real machine but exists as software running on AWS-managed hardware.

  • You manage: OS, packages, applications, runtime, firewall rules
  • AWS manages: Physical servers, power, cooling, hardware failures

7.2 Why EC2 Is Used in Enterprise

  • Elasticity: Scale up/down in minutes instead of weeks
  • Pay-as-you-go: Pay only for running time
  • Standardization: Same server image across dev, test, prod
  • Automation-friendly: Fully controllable via APIs, CLI, IaC
  • Foundation for DevOps: CI runners, app servers, workers, bastion hosts
What is the main benefit of "Elasticity" in EC2?

A: The ability to scale up or down in minutes to match demand, rather than waiting weeks for physical hardware to be procured.

7.3 How EC2 Works

When you launch an EC2 instance, AWS performs a controlled virtualization workflow:

  • AMI (Amazon Machine Image) defines the OS + base software
  • Instance Type defines CPU, RAM, networking performance
  • Hypervisor (Nitro system) isolates instances securely
  • EBS provides persistent block storage
  • VPC provides isolated networking

Conceptually:

Physical Server → AWS Hypervisor → EC2 Instance → OS → Application

7.4 EC2 Instance Lifecycle

  • Launch: Instance is created from an AMI
  • Running: CPU, memory, network are active (billable)
  • Stop: Instance halted, EBS retained (no compute cost)
  • Start: Instance resumes from stopped state
  • Terminate: Instance destroyed permanently

DevOps implication: Stop ≠ Terminate. Stopping preserves disks; terminating deletes compute.

What is the difference between "Stopping" and "Terminating" an instance?

A: Stopping pauses the instance (data is saved, no compute cost). Terminating permanently deletes the instance and its local data.

7.5 Steps to Create an EC2 Instance

  1. Select AMI (e.g., Amazon Linux, Ubuntu)
  2. Choose Instance Type (e.g., t3.micro)
  3. Configure Key Pair (for SSH access)
  4. Attach Security Group (firewall rules)
  5. Select Storage (EBS volume size/type)
  6. Launch instance

This produces a reachable VM with a public/private IP.

7.6 Accessing an EC2 Instance

  • Linux: SSH using key pair
  • Windows: RDP using decrypted password
  • Enterprise best practice: Use IAM Role + AWS Systems Manager (SSM), avoid SSH keys
ssh -i key.pem ec2-user@PUBLIC_IP

7.7 Instance Types

  • General Purpose (t, m): Web apps, CI agents
  • Compute Optimized (c): Build servers, batch jobs
  • Memory Optimized (r): Databases, caches
  • Storage Optimized (i): High I/O workloads

80% of enterprise workloads run on t, m, c families.

7.8 Storage with EC2 (EBS vs Instance Store)

  • EBS: Persistent, survives stop/start, snapshot-able
  • Instance Store: Temporary, data lost on stop/terminate

Rule: Production data → EBS

When should you use "Instance Store" vs "EBS"?

A: Use Instance Store for temporary caches (speed/ephemeral). Use EBS for databases and persistent data (safety/durability).

7.9 Networking & Security

  • VPC: Isolated virtual network
  • Security Groups: Stateful firewall (allow rules only)
  • NACLs: Subnet-level stateless filtering
  • Elastic IP: Static public IP for stable endpoints

Security Groups are attached to instances and control inbound/outbound traffic.

7.10 IAM Roles for EC2

An IAM Role allows EC2 to access AWS services securely without credentials in code.

  • EC2 → IAM Role → Temporary credentials
  • Used for S3 access, CloudWatch logs, ECR pulls

Enterprise rule: No access keys inside EC2.

Why should you avoid hardcoding AWS keys inside an EC2 instance?

A: It's a severe security risk. Instead, use an IAM Role attached to the instance to provide temporary, rotated credentials automatically.

7.11 Auto Scaling Groups (ASG)

ASG ensures the desired number of EC2 instances are always running.

  • Scale based on CPU, memory, or custom metrics
  • Replace unhealthy instances automatically
  • Foundation for high availability

DevOps mindset: Instances are disposable.

7.12 Load Balancing with EC2

  • ALB: HTTP/HTTPS traffic (most common)
  • NLB: TCP/UDP, high performance

Load balancers distribute traffic and remove failed instances from rotation.

7.13 Monitoring & Logging

  • CloudWatch Metrics: CPU, disk, network
  • CloudWatch Logs: Application and system logs
  • Alarms: Trigger scaling or alerts

Without monitoring, EC2 is blind infrastructure.

7.14 Pricing Models

  • On-Demand: Flexible, expensive
  • Reserved: Long-term predictable workloads
  • Spot: Cheap, interruptible (batch, CI)

Cost optimization is a DevOps responsibility.

7.15 EC2 in a DevOps Pipeline

  • CI runners (GitHub Actions self-hosted)
  • Application servers behind ALB
  • Bastion hosts for controlled access
  • Temporary build agents spun via automation

EC2 is often replaced by containers later, but it is the baseline skill.

7.16 Amazon Machine Image (AMI) Strategy

An AMI is a reusable snapshot of an EC2 instance (OS + software + config). Enterprises use AMIs to enforce consistency and speed up provisioning.

  • Golden AMI: Pre-hardened, patched, approved image
  • Immutable infrastructure: Replace instances instead of updating them
  • DevOps impact: Faster scaling, fewer configuration drifts
What is a "Golden AMI"?

A: A pre-configured, security-hardened snapshot of a server used to launch identical instances rapidly, ensuring consistency and compliance.

7.17 User Data & Bootstrapping

User Data is a script executed automatically at first boot. It is the primary mechanism to configure EC2 without manual login.

Used for installing packages, pulling code, starting services.

#!/bin/bash
yum update -y
yum install -y nginx
systemctl start nginx

Key idea: No manual setup after launch.

7.18 AWS Systems Manager (SSM) for EC2

SSM allows managing EC2 instances without SSH or open ports.

  • Session Manager: Secure shell-less access
  • Run Command: Execute commands remotely
  • Patch Manager: Automated OS patching

Enterprise rule: Prefer SSM over SSH.

7.19 EC2 Placement Concepts

Placement affects availability and performance.

  • Availability Zones: Fault isolation
  • Placement Groups:
    • Cluster: Low latency, high performance
    • Spread: Maximum fault tolerance

Used in high-availability and low-latency systems.

7.20 Resource Tagging

Tags are key–value metadata attached to EC2 resources.

  • Cost allocation
  • Ownership tracking
  • Automation and cleanup

Common tags: Env, Owner, Project, CostCenter.

7.21 EC2 Backup & Recovery (Snapshots)

EBS snapshots are point-in-time backups stored in S3.

  • Used for disaster recovery
  • Can create new volumes or AMIs
  • Automated via lifecycle policies

Snapshots enable rapid rollback.

7.22 EC2 vs Containers vs Serverless

  • EC2: Full control, long-running workloads
  • Containers (ECS/EKS): Managed orchestration
  • Serverless (Lambda): Event-driven, no servers

EC2 is the foundation layer beneath all others.

7.23 Common Enterprise Failure Patterns

  • Hardcoding credentials inside instances
  • Manual configuration after launch
  • Single instance without load balancer
  • No monitoring or alarms

These failures directly map to outages.

7.24 EC2 Security Hardening

  • Minimal inbound rules (principle of least privilege)
  • IAM roles instead of keys
  • Disable password-based SSH
  • Regular patching via SSM

Security is a continuous process, not a setup step.

7.25 When NOT to Use EC2

  • Highly spiky event-based workloads → Lambda
  • Pure container workloads → ECS/EKS
  • Short-lived background jobs → Spot + batch
Why is "Re-architecting" important?

A: Simply lifting and shifting applications to EC2 might not yield cost or performance benefits. Choosing the right architecture (like Serverless or Containers) is crucial.

7.26 AWS IAM (Identity and Access Management) Basics

IAM is the security gatekeeper of AWS. It controls who can access AWS resources (Authentication) and what they can do (Authorization). It is global (not region-specific).

  • Users: Specific people or services (long-term credentials).
  • Groups: Collections of users. Permissions applied to a group apply to all members.
  • Roles: Temporary identities with specific permissions. Defined for services (like EC2) or federated users. This is the secure way for EC2 to access other services.
  • Policies: JSON documents that define permissions (Allow/Deny actions on Resources).

7.27 Amazon S3 (Simple Storage Service) Basics

Start managing storage without worrying about servers. S3 is object storage designed to store and retrieve any amount of data from anywhere.

  • Buckets: Logical containers for objects (files). Names must be globally unique across all AWS accounts.
  • Objects: The actual files stored. Each object is identified by a unique "Key" (path).
  • Durability: Designed for 99.999999999% (11 9s) durability.
  • Scalability: Unlimited storage capacity.

7.28 Hands-On: Integrating EC2 with S3 using IAM

A common enterprise requirement is for an application on EC2 to read/write data to S3. Never manage this by hardcoding Access Keys and Secret Keys in your code.

Step 1: Create an IAM Role

  1. Go to the IAM Console → Roles → Create role.
  2. Select Verified Entity: AWS Service.
  3. Choose the Use Case: EC2.
  4. Add Permissions: Search for and select AmazonS3FullAccess (For production, use a more restrictive custom policy).
  5. Name the role (e.g., EC2-S3-Integration-Role) and Create.

Step 2: Launch EC2 with the Role

  1. Go to EC2 Console → Launch Instance.
  2. Configure your OS (AMI) and Instance Type as usual.
  3. In Advanced Details, locate IAM Instance Profile.
  4. Select the role you created: EC2-S3-Integration-Role.
  5. Launch the instance.

(Tip: If the instance is already running, go to Actions → Security → Modify IAM Role to attach it.)

Step 3: Verify Integration

  1. SSH into your EC2 instance.
  2. Use the AWS CLI (pre-installed on Amazon Linux) to list your buckets:
aws s3 ls

If successful, you will see a list of your S3 buckets. The EC2 instance securely "assumed" the role to get temporary credentials.
